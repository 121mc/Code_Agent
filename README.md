# Lightweight Terminal Coding Agent

Node.js and TypeScript CLI for a lightweight local coding agent.

https://github.com/121mc/Code_Agent

## Install

Requirements:

- Node.js 20.19.0 or newer.
- npm 10 or newer.
- A Git workspace for the best diff output. Non-Git workspaces still work for files modified during the current task.

From a source checkout:

```bash
npm ci
npm run build
npm install -g .
code-agent --help
```

From a packaged npm tarball:

```bash
npm ci
npm pack
npm install -g ./lightweight-terminal-coding-agent-0.1.0.tgz
code-agent --help
```

Primary distribution form: npm package tarball generated by `npm pack`. The package exposes the `code-agent` executable through the `bin` field in `package.json`.

## Runtime Configuration

The agent reads OpenAI-compatible settings from environment variables or `.code-agent/config.json`:

- `CODE_AGENT_BASE_URL`
- `CODE_AGENT_API_KEY`
- `CODE_AGENT_MODEL`

Safer local setup is to create `.code-agent/config.json` on the target machine. This file is ignored by Git and must not be committed:

```json
{
  "baseURL": "https://api.openai.com/v1",
  "apiKey": "replace-with-your-own-key",
  "model": "gpt-4.1-mini"
}
```

Environment variables are also supported, but they are visible to the current process and can leak through shell history if exported directly. The `/config` command only prints a masked key.

## Usage

```bash
code-agent
code-agent "change this function to handle empty arrays"
npm run dev -- "change this function to handle empty arrays"
```

Built-in commands:

- `/help`: show available commands.
- `/init`: create an initial `Claude.md`.
- `/diff`: show current changes.
- `/status`: show current task state.
- `/config`: show active model configuration without revealing secrets.
- `/exit`: exit the REPL.

The generated `lightweight-terminal-coding-agent-0.1.0.tgz` can be copied to another machine and installed with:


Known limits:

- Tested with Node.js 20.19.0+ on Windows and Linux CI.
- The CLI is source-local and does not include a hosted service or Web UI.
- The LLM provider must expose an OpenAI-compatible chat completions endpoint.
- `.code-agent/config.json`, `.env*`, `node_modules/`, `dist/`, and generated worktrees are not part of the committed source.

## Security Boundary

- The project never requires real API keys in the repository.
- `.code-agent/config.json` and `.env*` are ignored by Git.
- `/config` masks the API key before display.
- File tools restrict reads and edits to the startup workspace and require confirmation for sensitive targets such as `.env*`, lockfiles, CI files, and private key material.
- Command tools allow common test, lint, and build commands automatically, require confirmation for unknown or network/dependency commands, and block destructive command variants.

## Directory Structure

- `src/`: CLI, agent loop, LLM client, permissions, session state, and tool implementations.
- `tests/`: Vitest unit and integration tests, including mock-LLM agent flows.
- `SPEC.md`: project specification produced from Superpowers brainstorming.
- `PLAN.md`: implementation plan and task commit table.
- `SPEC_PROCESS.md`: brainstorming, planning, and cold-start process record.
- `AGENT_LOG.md`: task-by-task agent workflow evidence.
- `REFLECTION.md`: final reflection report.
- `.github/workflows/ci.yml`: GitHub Actions test/build workflow.
- `.gitlab-ci.yml`: GitLab-compatible `unit-test` job for course submission checks.

## CI

CI runs dependency installation, unit tests, and TypeScript build:

- GitHub Actions job id: `unit-test`.
- GitLab CI job name: `unit-test`.
- Required commands: `npm ci`, `npm test`, `npm run build`.

## Manual Acceptance

1. Create a small TypeScript sample project in a temporary directory.
2. Add a `Claude.md` with project conventions and test commands.
3. Configure `CODE_AGENT_BASE_URL`, `CODE_AGENT_API_KEY`, and `CODE_AGENT_MODEL`, or create `.code-agent/config.json` on that target machine.
4. Run `code-agent "change parseUser so empty input returns anonymous"`.
5. Confirm the agent prints a plan, searches and reads relevant files, applies one small edit, runs an allowed test or build command when requested by the model, prints changed files, and prints a final one-shot diff. Current diffs are also available through `/diff` in Git workspaces or through a model-invoked `diff` tool observation during the task.
